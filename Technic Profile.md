# 构妙 LiveCompose 中期答辩 PPT 方案

## 一、设计动机与优势

### 1.1 项目的动机是什么？解决了现有拍照方案的哪些痛点？

我们观察到，虽然现在的智能手机拍照功能很强大，但很多普通用户并不了解专业的摄影构图知识，比如“三分法”或者如何突出主体。他们拍出的照片常常构图平庸，或者抓不住重点。现有的相机应用大多只提供静态的辅助线，比如九宫格，但这并不能主动告诉用户“应该怎么移动手机才能构图更好”。用户仍然需要自己判断和调整，这既费时又容易错过精彩瞬间。我们的项目动机就是解决这个痛点，我们希望通过 AI 技术，实时分析画面内容，并主动引导用户移动手机，轻松获得构图更优良的照片，甚至在对齐时自动完成拍摄，让好照片的诞生更加“智能”和“轻松”。

### 1.2 产品的目标用户与核心目标是什么？用户通过产品能获得哪些提升？

我们的目标用户主要是那些希望提高手机摄影水平，但又缺乏专业摄影知识的广大普通用户，比如喜欢在社交媒体分享生活的年轻人，或者希望为家人拍出更好照片的父母。产品的核心目标是成为一个“智能构图助手”，通过实时反馈和引导，帮助用户在拍摄当下就完成出色的构图。用户通过使用我们的产品，最直接的提升就是照片的构图质量会显著改善，画面主体更突出，视觉效果更专业。同时，由于引导是实时的，用户可以在拍摄过程中潜移默化地学习到一些构图技巧。对于开启了自动拍照功能的用户，他们还能更专注于捕捉对象的表情和动作，将按下快门的时机交给应用判断，从而可能捕捉到更自然的瞬间。

### 1.3 对比同类 APP，项目的主要创新点和优势？

目前市场上的相机应用，要么是专注于滤镜和后期处理美颜相机，要么是提供静态的构图辅助线的系统相机。我们的主要创新点在于“实时动态引导”和“智能追踪”。我们不仅通过 AI 模型分析画面，确定一个理想的构图区域，更关键的是，我们结合了设备的陀螺仪数据实现了一个物理上更准确、体验更流畅的追踪系统，实时在屏幕上显示目标点，引导用户移动手机去对齐。这种“检测-追踪-引导-拍摄”的闭环是我们的核心优势。相比于简单的网格线，我们的引导是动态的、交互式的，这让获得一张构图良好的照片不再需要复杂的思考和调整，操作门槛大大降低，实现摄影的普惠化与大众化

## 二、技术实现

### 2.1 项目的整体系统架构图？说明各个模块的功能与交互方式

**系统架构图：**

![Snipaste_2025-11-08_21-43-16](./LiveCompose 中期答辩 PPT 方案.assets/Snipaste_2025-11-08_21-43-16.jpg)

我们在 ios 平台上设计应用，并将 LiveCompose 构图模型内置于应用中，系统采用了四层设计

我们的架构基础是第一层，也就是数据采集层。这一层直接和硬件打交道。它主要包含两个模块：一个是 `CameraManager`，它利用苹果的 AVFoundation 框架来管理相机，负责捕捉实时的视频帧数据流 和拍摄最终的照片；另一个是 `MotionStabilityMonitor`，它通过 CoreMotion 框架持续获取设备的陀螺仪和加速度数据。这一层的工作就是为上层提供原始图像和运动数据。

拿到原始数据后，就进入了第二层，智能决策与追踪层。这一层是实现智能化的关键。它包括了我们的核心 `AdacropModel`，这是一个通过 CoreML 运行的强化学习模型，它负责分析视频帧，计算出最佳的构图建议。同时，这一层还有 `BoxCenterManager`，它获取第一层的陀螺仪数据，实现一个基于物理的追踪模型。这个追踪器不仅能计算目标位置，还带有一个“磁性吸附”功能，当用户接近中心点时能自动吸附，帮助稳定对齐。

第三层是核心协调层，它是整个应用的大脑。这一层只有一个关键模块，`CaptureViewModel`。它负责管理一个复杂的状态机，比如“等待稳定”、“正在检测”、“准备拍照”等。它从第一层拿到数据，指挥第二层在设备稳定时才启动 `AdacropModel` 进行分析。当模型给出结果后，它把目标点交给 `BoxCenterManager` 去追踪。它还持续监控追踪状态，一旦发现“已对齐”，就会触发自动拍照逻辑，并根据当前状态生成引导文字给用户看。所有来自界面的操作，比如变焦或切换镜头，也都由它来响应和调度。

最上面是第四层，视图与交互层，也就是用户直接看到和操作的界面。这一层完全由 SwiftUI 构建。它包含了 `ContentView` 作为主视图，一个 `CameraPreview` 视图来显示底层的相机预览，以及 `UserGuidanceView` 和其他 UI 组件，如按钮和变焦环。这一层的所有视图都订阅 `CaptureViewModel` 的状态。当 `ViewModel` 更新数据时，例如追踪点位置或引导文字变化时，界面会自动刷新，同时，用户的点击操作也会被传递回 `ViewModel` 去执行。

### 2.2 选取了什么模型，如何进行训练，数据集如何搭建，模型的参数等等

#### 1. 模型架构

- **主干网络** （Backbone）：torchvision ResNet50（使用ImageNet 预训练），去掉 fc（self.backbone.fc = Identity），特征维度 2048。用于提取图片的高层语义特征，供后续分支使用。
  - 输入：`[B, 3, 224, 224]` 图像张量为 3×224×224，将最长边缩放到224，另一边空白部分用黑色填充，**保持原始比例**
  - 输出：特征向量 `[B, 2048]`
- **Actor 分支**
  - **结构**：MLP（2048+4 → 1024 → 512 → n_actions）。
  - **输入**：图片特征 `[B, 2048]` + 当前裁剪框状态 `[B, 4]`（归一化的 (cx, cy, w, h)，分别是裁剪框左上角坐标和宽高，均除以原图尺寸，范围在 0~1）。
  - **输出**：动作概率分布 `[B, n_actions]`。
  - **作用**：决定 agent 在当前状态下选择哪个动作（比如移动、缩放、停止等）
- **Critic 分支**
  - **结构**：MLP（2048+4 → 1024 → 512 → 1）。
  - **输入**：同 Actor。
  - **输出**：当前状态的价值估计 `[B, 1]`。
  - **作用**：用于强化学习中的优势估计和基线，即当前状态有多好
- **BBox Head 分支**
  - **结构**：MLP（2048 → 512 → 4）。
  - **输入**：仅图片特征 `[B, 2048]`。
  - **输出**：预测裁剪框参数 `[B, 4]`。
  - **作用**：只在监督预训练阶段用，回归真实裁剪框。

训练数据与环境

- 数据来源：config.data.train_json / val_json（每条记录含 rec['img'] 路径）。
- 环境（CropEnv）：
  - 状态：裁剪后的图像张量 + 归一化几何向量。
  - 奖励：基于 NIMA 美学模型的分数变化（tanh 形状）+ 访问/重复/无效动作惩罚；stop 动作有额外设计。
  - NIMA 评分器：A800SharedNIMAScorer（PyTorch）
    - 架构按权重路径自动选 Inception v3 或 ResNet50，末层输出 10 桶评分分布，期望值为得分（1~10）。
    - 实际评分每 real_score_interval=5 步计算一次，其余用几何启发估计；含缓存与限幅。
    - 权重路径：nima.weights_path（示例 ./NIMA/weights/nima_inception_pytorch.pth）。若缺失则不会加载自定义权重，仅用 ImageNet/随机权重，效果会差。

#### 2. 训练流程

##### 2.1 监督预训练

- **训练部分**：只训练 backbone 和 bbox_head

- 流程 ：

  1. 输入图片，backbone 提取特征。
  2. bbox_head 回归裁剪框参数。
  3. 用 MSELoss 和真实框做监督训练。
  4. 训练结束后，把 backbone 和 bbox_head 的权重迁移到完整 ActorCritic 模型。

- **目的**：让 backbone 学会裁剪相关特征，bbox_head学会预测合理裁剪框，为后续 RL 提供更好的初始化。

##### 2.2 强化学习

- **训练部分**：用完整的 ActorCritic（backbone + actor + critic）。

- 环境 CropEnv

  - 状态组成：

    - 图像裁剪张量 img_tensor: ` [3, img_size, img_size]`，由当前 box 裁剪 + resize 得到
    - 几何状态 state_tensor: `[x/iw, y/ih, w/iw, h/ih]`

  - 动作集合 self.actions（共 11 个）：

    left, right, up, down, zoom_in, zoom_out, wider, narrower, taller, shorter, stop 

  - 步进逻辑：

    1. 根据动作修改 box（平移 / 缩放 / 改变宽高）。

    2. 边界裁剪 并 判断最小尺寸约束

    3. 评分：new_score = _get_score(new_box)
       - 若当前步数满足间隔: real（调用 NIMA）
       - 否则：几何估计 _estimate_score_safe

    4. 奖励：
       - 分数差 score_diff = new_score - prev_score
       - base_reward = tanh(score_diff * 2.0)
       - 惩罚 penalty：
         - 重复访问同一裁剪：visit_penalty
         - 连续完全未改变（no-op）：no_op_penalty
         - 过度重复到 max_repeat：repeat_penalty
       - stop 动作特殊：奖励 = (final_score - prev_score)*2.0 +（早停惩罚或适时停奖励）

    5. 更新 prev_score, best_score, step_count；返回 (img_tensor, state_tensor), reward, done, info

  - 重置 reset()

    - 随机初始化一个可行 box

    - 初始裁剪评分（真实 NIMA 调用）

    - 清理缓存过大时的 score_cache

    - 返回初始状态

![截屏2025-11-09 15.13.07](LiveCompose 中期答辩 PPT 方案.assets/截屏2025-11-09 15.13.07.png)

- 真实 NIMA 评分比较耗时，为保证训练进度做了以下处理：

  - reset() 会调用 NIMA 得到初始分（init_score）。

  - 每次需要评分时，env._get_score 会先查缓存 score_cache；若命中即不调用 NIMA。

  - 否则按间隔决定是否真实调用：当 self.step_count % self.real_score_interval == 0 时调用 _safe_nima_score（即调用 NIMA）；否则用几何估计 _estimate_score_safe。

  - stop 动作也通过 _get_score 获取分数，因此是否调用 NIMA 同样受 real_score_interval 和缓存约束。

  - real_score_interval 从 config.yaml 的 nima.real_score_interval 读取，默认值为 5（即大约每 5 步一次真实评分）

- Actor 和 Critic 的协作
  - Actor 通过 policy_loss，鼓励在优势 adv > 0 的状态下提高对应动作概率，抑制 adv < 0 的动作。

  - Critic 通过 value_loss 逼近 returns，目标是预测未来累计奖励期望

  - Critic 提供 baseline，减少 Actor 目标的高方差，更精确的价值 → 更稳定的优势 → Actor 更新更有效。同时，Actor 改善策略 → 收集到的 trajectories 奖励分布更高 → Critic 训练目标更容易。

### 3. 当前架构的合理性

####  3.1 为什么选择强化学习？

图片裁剪不是一次性输出最终框，而是多步微调（移动 / 缩放 / 调整比例 / 停止），天然是 Markov 决策过程。NIMA 得分、本地几何启发（黄金比例、面积偏好）对最终框“质量”提供信号，但这些评分函数与奖励不可微，不可直接端到端反传；策略梯度可绕过可微性限制。而且未来可以更加方便的扩展动作空间（比如旋转）

#### 3.2 为什么模型选择当前结构？

- Actor–Critic 优于纯策略梯度：Critic 估计价值降低方差，加速收敛，尤其奖励噪声大（几何估计 + 间隔真实评分）。

- 分离 bbox_head 预训练：先回归标注或伪标注裁剪框使 backbone 学会“构图相关特征”（主体、分割、线条结构），减少 RL 冷启动瞎探索。

- 使用 ResNet50：有较强泛化的高层特征，对人物/风景/物体类别具有鲁棒性，避免在 RL 早期陷入局部随机梯度，而且后续蒸馏到 MobileNet 难度也不大 。

- Dropout + MLP：简单、推理成本低，方便后续裁剪为移动端轻量化版（替换 蒸馏后的backbone 即可）。

  

#### 3.数据集构建

![image-20251108212400447](./LiveCompose 中期答辩 PPT 方案.assets/image-20251108212400447.png)

我们的数据集制作流程受AAAI 24论文《Learning Subject-Aware Cropping by Outpainting Professional Photos》的启发

首先，我们需要准备一组原始数据，它是来自专业摄影师的摄影集，具有良好的构图结构与美学评分。

我们采用**双模型工作流**：

**语义理解模型：`BLIP Image Captioning (base)`**

- **角色：**告诉扩图模型应该扩什么？
- **选型理由：** 扩图不仅是填充像素，更是要对内容进行理解，这样结果才能和原图更一致。BLIP 模型能为原始图片生成准确的语义描述，为后续的生成模型提供高质量的上下文指导。

**内容生成模型：`Stable Diffusion v2 Inpaint`**

- **用途：** 扩图。
- **选型理由：** Inpaint（修复）模型是实现 Outpaint（扩图）的标准方案。`SD v2 Inpaint` 擅长根据给定的图像、掩码和提示词，在指定区域生成与上下文高度融合的新内容。

**自动化工作流：**

1. **生成扩图提示词**
   - 正面提示词：由于我们希望扩图得到的图片和原图在内容上保持一致性，因此我们的正面提示词必须和原图密切相关。
     - 每张原图被送入BLIP模型，生成与其相关的基础描述。
     - 将这个描述嵌入到一个“任务指令”模板中，形成最终的正面提示词。
   - 负面提示词：
     - 负面提示词用于在生成过程中抑制低质量特征的出现
     - 包含常见的AI生成缺陷与伪影项，如 `frame, border, text, watermark` 等
2. **多样化扩图生成**
   - 为增加数据集的多样性，我们没有采用固定的**扩展比例**、**提示词相关性**和**迭代步数**。
     - 脚本在 `min_ratio` 和 `max_ratio`（10% 到 47.5%）的配置范围内，为原图的**上、下、左、右**四个方向**独立地、随机地**生成一个扩展边距，从而生成具有不同留白和重心的扩展样本，极大丰富了数据的多样性。
     - 提示词相关性 和 迭代步数的组合变化，会产生不同相合程度和细节程度的图像，提高了数据的异质性。

生成图片后，我们要对生成的数据质量进行控制，共分三个步骤：

1. **边框伪影自动检测**
   - **问题：** Inpaint 模型有时会在原图和生成区域的接缝处产生一个可见的“画框”伪影。
   - **方案：** 在图像保存前，脚本会调用 **Canny 边缘检测** 。如果该函数在图像中检测到高强度的、类似边框的边缘，系统会判定此次生成为“失败”，**自动丢弃该样本**，并（在循环中）尝试下一次生成。
2. **原图一致性校验**
   - **问题：** Inpaint 模型可能污染了被Mask保护的原始图像区域。
   - **方案：** 对于每一张成功生成的扩展图，脚本会**反向裁剪**出原图所在 `bbox` 区域的像素，并将其另存。我们只需对比裁切后图片与原图的相似度，即可快速验证模型是否严格保护了原始图像的区域。
3. **人工快速筛选**
   - **问题：**对每个图，我们可以选择扩展N个版本，各个版本是否可采用最好需要人工审核以提高数据集的质量。
   - **方案：** 脚本会为每张原图自动拼接一张预览图，该网格包含了 N 个成功的生成版本 + 1 张原图（带白边）的对比。这使我们能以“一图流”的方式，高效地对生成结果进行人工抽检和评估。

最终我们会得到【扩展后的图片，原图的坐标框】的二元组数据集。

### 2.3 如何处理分析实时相机视频流？关键的技术选型是？

在现代移动应用开发中，实时视频流处理是计算机视觉、增强现实和智能拍摄应用的核心能力。LiveCapture 作为一款智能构图相机应用，其核心挑战在于如何以最低的延迟、最高的性能处理每秒数十帧的视频流，同时还要运行复杂的 AI 模型进行场景分析。本文将深入探讨 LiveCapture 如何构建其视频流处理管线，以及在技术选型上做出的关键决策。

视频流处理采用了**多层级管线架构**，从硬件层到应用层形成了清晰的职责分离，各层之间通过明确的接口通信

- **可测试性**：每层都可以独立进行单元测试
- **性能优化**：可以针对性地优化特定层的性能瓶颈
- **可维护性**：清晰的职责划分使代码易于理解和维护

```
CaptureViewModel.handleSampleBuffer (业务处理)
    ├─ 稳定性检查 (MotionStabilityMonitor)
    ├─ 3:4 构图裁切 (CIImage & CIContext)
    ├─ Adacrop 模型检测 (AdacropModel)
    └─ 追踪更新 (BoxCenterManager)
        ↓
SwiftUI 视图更新 (@Published 属性)
    ↓
用户界面渲染
```

---

在 iOS 平台上，视频捕获有多种方案可选，包括：

- **UIImagePickerController**：系统自带的拍照界面，功能有限，无法实现追踪。
- **AVFoundation**：底层框架，提供完全的控制权，原生开发高效。
- **第三方框架**（如 GPUImage）：提供高级滤镜和特效，但适配难度高。

而我们为了尽可能使用原生的组件，降低硬件性能损耗，使用 **AVFoundation** 框架，同时基于项目特点设计了串行队列保证线程安全。CameraManager 会等待 CoreMotion 传入稳定信号后，调用AVCaptureVideoDataOutput 拉取出最新的一帧视频帧传入 AdacropModel，这是我们从刚才介绍的 Adacrop Pytorch 版本转译为 CoreML 的轻量化 Student 模型，继承了教师端模型的参数和训练效果，拥有更高效的处理速度。模型返回 Crop_box，调用追踪系统进行 UI 引导，最后调用 CoreImage 组件输出高分辨率图像存入相册。

### 2.4 构图建议如何生成与引导用户？UI 的设计是怎么样的

此处是要介绍 UI 的设计以及陀螺仪的追踪模块

构图建议的生成和引导是一个两阶段的过程，首先通过复杂的 AI 分析找到一个“好”的构图，然后通过一种轻量级的方式引导用户去“复现”这个构图。UI 设计直观地将这个过程可视化。

当我们得到了一个好的构图，重中之重便是要寻找一条有效的方案引导用户将画面移动到构图的模样，因此我们开发了一套基于用户的追踪与 UI 引导系统。其本质是一个多维度传感器融合系统，需要实时协调三个关键模块：

- **陀螺仪追踪** - 通过设备姿态变化计算目标位移
- **稳定性监控** - 评估设备运动状态确保检测和拍照时机
- **UI 的显示** - 引导用户移动设备并在合适的时机正确拍摄

因此我们围绕三个递进目标展开：

#### 目标一：**状态可视化（State Visibility）**

用户需要知道系统处于哪个阶段，是在等待、检测、还是准备拍照。这通过**动态文字引导**和**进度阶段映射**实现。

#### 目标二：**操作指引（Action Guidance）**

在每个阶段，用户需要明确知道"我应该做什么"。系统通过**情境化的提示文字**和**画面视觉标记**告诉用户当前的行动方向。

#### 目标三：**反馈闭环（Feedback Loop）**

用户的每个动作（移动设备、保持稳定）都应该得到**即时可见的反馈**，这通过**动态颜色渐变**和**触觉震动**实现。

接下来我将详细的叙述其中的关键节点：

第一我们设计了一个基于物理逻辑的追踪算法，其核心逻辑是为构图模型返回的构图选取中心点，并为该点添加实时的位置运算。当用户移动手机时，追踪点会依照物理逻辑在画面中相应产生位移，具象到实际效果便呈现出点能追踪物体的效果。而当追踪点与中心点接近时，我们为追踪点增加磁性吸附机制，这使得两点能像磁铁一样有着吸附关系，用户的潜意识会自然想要将两个点对齐，而当两个点对齐后，我们将其变绿和锁定，并向 CameraManager 发送拍摄信号，一次智能构图的流程就完成了。

第二，仅靠简单的追踪点来引导用户还是略显单调，我们通过画面上方的 GuidanceBar 和 Haptic Engine 为用户同时提供文字与触觉反馈引导

| 事件                 | 反馈类型      | 用户感知               |
| -------------------- | ------------- | ---------------------- |
| 点击魔术棒开启流水线 | `success()`   | "功能已激活"           |
| 检测到目标区域       | `success()`   | "识别成功"             |
| 追踪点对齐中心       | `focusLock()` | "对准了！"（双击震动） |
| 追踪点失去对齐       | `warning()`   | "偏离了"（警告震动）   |
| 拍照触发             | `capture()`   | "快门按下"（双段震动） |
| 照片保存成功         | `success()`   | "任务完成"             |
| 检测到大幅运动       | `warning()`   | "需要重新稳定"         |

| 阶段     | 引导文字               | 设计意图                       |
| -------- | ---------------------- | ------------------------------ |
| 空闲     | 点击魔术棒开启流水线   | 引导用户点击右下角的魔术棒     |
| 启动     | 正在启动相机           | 告知系统正在准备，减轻等待焦虑 |
| 等待稳定 | 请保持稳定             | 明确的动作指令（静止）         |
| 检测中   | 正在识别最佳构图...    | 解释延迟原因，传达智能感       |
| 追踪就绪 | 请将圆点移动到画面中心 | 具体的操作目标                 |
| 对齐成功 | 即将拍照，请保持稳定   | 确认成功 + 持续要求            |
| 拍照中   | 正在拍照...            | 避免用户重复触发               |
| 完成     | 照片已保存             | 积极反馈，任务闭环             |
| 错误     | 发生错误，请重试       | 友好的错误提示                 |

最后，如果一个画面本身就是很糟糕的角度和移动，不论我们的构图模型再强大也无法给出一个好的建议。因此我们为画面增加了基础的三分构图线辅助用户自然的将手机置平；同时为防止画面很乱，当设备移动速度过快我们也会重置拍摄流程。

### 3.3 当前项目完成度是如何？后续规划

**LiDAR 深度增强**

利用 iPhone Pro 的 LiDAR 传感器获取场景深度图，准确测量主体距离用于自适应增益。深度信息还能优化 Vision 检测——结合深度边缘提升人物与背景的分割精度，在复杂背景下更准确识别主体。深度数据的低延迟（< 10ms）适合实时追踪场景。

**云端协作与社区学习**

上传用户拍摄的照片（匿名化处理）到云端，训练全局美学模型。模型定期推送更新到设备，让系统不断学习新的摄影趋势和风格。社区优秀作品的构图参数可作为预设供用户选择，形成"站在巨人肩膀上"的创作辅助生态。
